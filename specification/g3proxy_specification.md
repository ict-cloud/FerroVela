# g3proxy Integration Specification

## Overview

The current implementation of Ferrovela relies on Cloudflare's `pingora` proxy framework. While `pingora` is highly performant, it is primarily designed as a reverse proxy. This introduces limitations for a client-side forward proxy like Ferrovela, particularly when handling full HTTP `CONNECT` tunneling and raw TCP/TLS stream proxying.

To address these limitations, we are replacing `pingora` with `g3proxy` (from Bytedance). `g3proxy` is an enterprise-level generic proxy solution with native support for:
- Forward proxy protocols (HTTP/1, SOCKS5).
- Transparent proxying (SNI Proxy, TCP TPROXY).
- Full `CONNECT` and `STREAM` support.
- Extensive authentication mechanisms.

This document outlines the architectural changes, focusing on Kerberos authentication for the initial prototype, and describes the benchmarking strategy.

## Architecture Replacement

### Current Architecture (`pingora`)
- **Server:** `pingora::server::Server` handles the main event loop and accepts connections.
- **Proxy Logic:** `FerroVelaProxy` implements `pingora::proxy::ProxyHttp`.
- **Limitations:** Limited native support for raw TCP streaming and complex multi-roundtrip authentication handshakes across proxy hops in a forward proxy scenario.

### Proposed Architecture (`g3proxy`)
- **Server Component:** Utilize `g3proxy`'s internal server configurations (e.g., `HttpProxy` or `TcpStream`) to bind to the local port (`127.0.0.1:3128`).
- **Escaper Component:** Use `g3proxy`'s "Escaper" abstractions to route traffic to the upstream proxy or directly to the target.
- **Resolver Component:** Integrate our existing PAC engine (`PacEngine`) into `g3proxy`'s routing mechanism or implement a custom resolver/escaper that queries the PAC engine dynamically.
- **Authentication:** Implement a custom authenticator or integrate our existing `KerberosAuthenticator` (using `libgssapi`) into `g3proxy`'s `Escaper` authentication pipeline.

## Feature Parity & Enhancements

1. **Full `CONNECT` and `STREAM` Support:**
   - `g3proxy` natively handles HTTP `CONNECT` requests, establishing a raw TCP/TLS tunnel between the client and the destination.
   - We will configure the local server as an `HttpProxy` that supports `CONNECT`.

2. **Kerberos Authentication (Prototype Focus):**
   - The primary challenge in forward proxying with Kerberos is the `Proxy-Authenticate: Negotiate` handshake.
   - We will inject the `Proxy-Authorization` header generated by our `libgssapi` wrapper into the upstream requests made by `g3proxy`'s Escaper.
   - The session state (GSS context) will be maintained across the 407 retry loops.

## Implementation Steps (Prototype)

1. **Configuration Mapping:** Map Ferrovela's `Config` to `g3proxy`'s internal configuration structures dynamically.
2. **Server Initialization:** Replace `pingora::server::Server` with `g3proxy` runtime initialization in `src/proxy/mod.rs`.
3. **Upstream Routing:** Implement a custom router that bridges `g3proxy` with `resolve_proxy` (which uses PAC/Exceptions).
4. **Kerberos Integration:** Bind the `KerberosAuthenticator` to the specific upstream escaper.

## Benchmarks

To validate the performance of the `g3proxy` implementation, we will use the `criterion` crate to benchmark:
1. **CONNECT Latency:** The time taken to establish a `CONNECT` tunnel through the proxy to a local echo server.
2. **STREAM Throughput:** Data transfer rates over established raw TCP streams.
3. **Kerberos Handshake Overhead:** Latency introduced by the initial GSSAPI negotiation and token generation.

The benchmarks will be located in the `benches/` directory and can be run via `cargo bench`.
